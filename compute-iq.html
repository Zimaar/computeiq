The Two Constraints

Inference sizing is governed by two independent constraints. The binding constraint (whichever requires more GPUs) determines your final cluster size:

  • VRAM constraint: The model weights must fit in GPU memory. This sets the minimum GPUs per replica.
  • Throughput constraint: The cluster must generate tokens fast enough to serve all concurrent users. This sets the number of replicas needed.

VRAM Sizing Formula

Step 1 — Model VRAM
Model VRAM (GB) = Parameters (B) × Bytes per Parameter

Bytes per parameter:
  FP32 = 4 B/param   FP16/BF16 = 2 B/param
  FP8/INT8 = 1 B/param   INT4/GPTQ/NF4 = 0.5 B/param

Example (70B, FP16): 70 × 2 = 140 GB
Step 2 — GPUs per Replica
Usable VRAM per GPU = GPU VRAM × 0.88 (12% OS/framework overhead)

GPUs per replica = ⌈ Model VRAM / Usable VRAM ⌉

Example (70B FP16, H100 80GB usable=70.4GB):
⌈ 140 / 70.4 ⌉ = 2 GPUs per replica

Throughput Formula

Step 3 — Required Throughput
Tokens per request = Input tokens + Output tokens
  (Input ≈ 80% of context, Output ≈ 20% of context)

Required throughput (tok/s) =
  Concurrent Users × Requests/User/Hour × Tokens/Request ÷ 3,600

Example (500 users, 120 req/hr, 32K context):
500 × 120 × 32,768 ÷ 3,600 = 546K tok/s
Step 4 — GPU Count
Effective tok/s per GPU = GPU baseline tps × model_size_factor × dtype_multiplier

Replicas needed = ⌈ Required throughput / Effective tok/s per GPU ⌉

Total GPUs (active) = Replicas × GPUs per replica
Total GPUs (with HA) = ⌈ Active GPUs × 1.25 ⌉  (25% buffer for failures + upgrades)
GPU Utilisation Target: Aim for 60–80% utilisation. Below 40% you are over-provisioned and wasting cost. Above 85% you are at saturation risk — a traffic spike will cause queuing and latency spikes.

🧠 Training Sizing

How to size a training cluster using the 6PD FLOPs rule, Chinchilla scaling laws, and correct per-GPU VRAM decomposition — validated against the H2LOOP 100B example.

6PD RuleChinchilla ScalingAdamW VRAM

The 6PD Rule (Total Compute)

The standard formula for the total FLOPs required to train a dense Transformer model is:

Total Compute (6PD Rule)
C = 6 × P × D

Where:
  C = Total FLOPs required
  P = Number of parameters (e.g., 100 × 10⁹ for 100B)
  D = Number of training tokens (e.g., 2 × 10¹² for 2T tokens)

Why 6? The forward pass ≈ 2PD FLOPs, backward pass ≈ 4PD FLOPs (2× forward).
Each MAC (multiply-accumulate) = 2 FLOPs.

H2LOOP 100B example:
6 × (100 × 10⁹) × (2 × 10¹²) = 1.2 × 10²⁴ FLOPs

Chinchilla Scaling Law

The Chinchilla paper (Hoffmann et al., 2022) established that compute-optimal training requires roughly 20 training tokens per parameter:

Chinchilla Optimal Tokens
D_optimal = 20 × P

For a 100B model: D = 20 × 100B = 2 Trillion tokens
For a 7B model: D = 20 × 7B = 140 Billion tokens
For a 70B model: D = 20 × 70B = 1.4 Trillion tokens

Chinchilla Ratio = D_actual / D_optimal
  0.8–1.5× = ✅ compute-optimal range
  <0.8× = under-trained (more data needed)
  >1.5× = over-trained on compute (diminishing returns)

Cluster Sizing Formula

GPU Count from Compute
Required cluster throughput = C / target_time_seconds (TFLOPs/s)

Effective TFLOPs per GPU = Peak TFLOPs × MFU
  MFU (Model FLOPs Utilisation): H100 typical = 38–50%
  H2LOOP example uses 45% → 989 × 0.45 = 445 TFLOPs/GPU

GPUs (compute) = ⌈ Required throughput / Effective TFLOPs/GPU ⌉

H2LOOP 100B, 60-day target:
Required = 1.2×10²⁴ / (60×86400×10¹²) = 231,000 TFLOPs/s
GPUs = ⌈ 231,000 / 445 ⌉ = 519 → rounded to 512 GPUs

Storage & Checkpointing

Checkpoint Size
Checkpoint = Weights (BF16) + Gradients (BF16) + AdamW (FP32)
  = P×2 + P×2 + P×12 = P × 16 GB total

For 100B model: 100 × 16 = 1,600 GB = 1.6 TB per checkpoint

To write in <30 seconds: need > 1,600 / 30 = ~54 GB/s write throughput
→ Requires parallel file system: Lustre, GPFS, or Weka
Checkpointing Strategy: Save every 500–1,000 steps (roughly every 2–4 hours). Keep 5 rolling checkpoints (corruption protection) plus permanent milestones at every 10% of training completion.

Agentic vs. Inference Sizing

Agentic platforms are more complex than simple inference because each "turn" involves multiple LLM calls, tool invocations, vector database lookups, and potentially nested sub-agent calls. The key multiplier is tool calls per turn:

Required LLM Throughput
Tokens per turn = Context tokens + 25% output tokens

Required tok/s =
  Sessions × Turns/session × Tool_calls/turn × Tokens/turn
  ÷ 3,600 × Platform_overhead_factor

Platform overhead factors:
  RAG: 1.3×   MCP: 1.5×   Multi-Agent: 2.2×
  Code Agent: 1.8×   Voice: 1.6×   Enterprise: 2.8×

Infrastructure Components

ComponentPurposeSizing Rule
LLM GPUsCore language model inferenceFrom throughput formula above + 25% HA buffer
Embedding GPUsConvert text/queries to dense vectors~50K tok/s per A10/L4 GPU. Scale by embed throughput needed.
Reranker GPUsCross-encoder relevance scoring (RAG, Enterprise)~500 session/s per GPU. Used in RAG and Enterprise platforms.

Latency Budget

Turn Latency Breakdown
TTFT (Time to First Token) = Context tokens / (tok/s × num_turns)
Generation latency = Output tokens / tok/s
Tool roundtrip = 200ms × tool_calls_per_turn

Total turn latency = TTFT + Generation + Tool roundtrips

SLA feasibility:
  Total latency ≤ SLA target → ✅ Feasible
  Total latency × agent_depth ≤ SLA → ✅ Nested agents feasible
Agent Nesting: A depth-3 multi-agent system (orchestrator calls 3 specialist agents, each of which makes sub-calls) multiplies latency by roughly 3×. Ensure your SLA budget accommodates the full depth. Use async parallel calls where possible.

How TaaS Sizing Works

IQ Sizer uses measured GPU throughput benchmarks (MLPerf, TRT-LLM, vLLM) combined with workload-specific peak-to-average analysis to produce a grounded, defensible cluster recommendation — not rule-of-thumb estimates.

Scientific Sizing Pipeline
Step 1 — Average Throughput
avg_tok/s = Monthly Tokens ÷ (30 × 86,400 seconds)

Step 2 — Peak Throughput (PAR)
peak_tok/s = avg_tok/s × Peak-to-Average Ratio
Chat=4× | Code=6× | Batch=1.2× | Fraud=8× | RAG=3× | Voice=5×

Step 3 — Effective GPU Throughput (from benchmarks)
gpu_eff_tok/s = published_benchmark_tps[gpu][model_size][dtype]
e.g. H100 SXM FP8 8B = 8,000 tok/s   L40S FP8 8B = 3,600 tok/s

Step 4 — GPU Count
GPUs = ⌈ peak_tok/s ÷ gpu_eff_tok/s ⌉ × SLA_factor
SLA factors: 99%=1.15×   99.9%=1.25×   99.99%=1.40×

Step 5 — VRAM Validation
model_vram = params_B × bytes_per_param(dtype) + 10% KV-cache overhead
If model_vram > gpu_vram → auto-upgrade GPU tier or suggest multi-GPU

Step 6 — Cost and Economics
monthly_cost = GPUs × cheapest_cloud_rate(from live data) × 730 hrs
cost_per_M_tokens = monthly_cost ÷ (monthly_tokens ÷ 1,000,000)
min_price = cost_per_M ÷ (1 − target_margin)

💰 CAPEX vs OPEX

When to buy hardware versus when to rent cloud GPUs — the GPU Pricing Playbook 2026 framework for build-vs-rent decisions.

The Break-Even Formula

Break-Even Calculation
CAPEX = GPU_units × unit_price + BOM_cost
BOM cost (8-GPU server): $9K–$18K+ (networking, chassis, power, cooling)

OPEX/month = GPUs × cheapest_cloud_rate × 730 hours

Break-even (months) = CAPEX / OPEX_per_month

Decision rule (Playbook 2026):
  ≤ 18 months → ✅ Buy (CAPEX preferred)
  18–36 months → ⚠️ Situational (hybrid or lease)
  > 36 months → ❌ Rent (OPEX preferred)

Utilisation Rule

>30% GPU utilisation → Buy. At sustained 30%+ utilisation, owned hardware pays back within the hardware lifecycle (3–5 years) and is almost always cheaper than cloud over that period.
<25% GPU utilisation → Rent. At low utilisation, cloud spot pricing (especially providers like Salad at $0.09–$0.14/hr for RTX 4090/3090) beats the CAPEX model. Don't buy hardware to leave it idle.

Step 2 — VRAM Formula
weights_GB = params_B × bytes_per_param
  FP32: 4 B/param · FP16/BF16: 2 B/param

kv_cache_GB ≈ 0.5–8 GB (context-dependent)
overhead_GB ≈ 10–15% of weights

total_vram = weights + kv_cache + overhead
Step 3 — GPU Count
usable_vram = gpu_vram × 0.88

num_gpus = ⌈total_vram ÷ usable_vram⌉

MFUModel FLOPs Utilisation — ratio of actual to theoretical peak FLOPs. H100 achieves 38–50% MFU in practice due to memory bandwidth bottlenecks, communication overhead, and kernel efficiency.
6PD RuleC = 6 × P × D. The standard formula for training FLOPs in a dense Transformer. Forward ≈ 2PD FLOPs, backward ≈ 4PD FLOPs.
Chinchilla RatioD_actual / D_optimal where D_optimal = 20 × P. Ratio of 1.0 is compute-optimal. <0.8 is under-trained, >1.5 shows diminishing returns.
VRAMVideo RAM — GPU memory. Different from system RAM (DRAM). All model weights, activations, and optimizer states must fit in VRAM during training/inference.
KV CacheKey-Value cache storing attention states for each token. Grows linearly with context length and batch size. Major VRAM consumer in long-context inference.
TP (Tensor Parallel)Splits individual weight matrices across multiple GPUs. Requires high-bandwidth NVLink. Reduces per-GPU memory proportional to TP degree.
PP (Pipeline Parallel)Different model layers on different GPUs. Creates a pipeline of micro-batches across GPUs. Requires InfiniBand for cross-node PP.
DP (Data Parallel)Multiple identical model replicas, each processing different data batches. Gradients all-reduced at step boundary. Most scalable strategy.
ZeRO-3DeepSpeed Zero Redundancy Optimizer Stage 3. Shards model parameters, gradients, AND optimizer states across all GPUs. Maximum memory efficiency.
FSDPFully Sharded Data Parallel. PyTorch's equivalent to ZeRO-3. Shards all model state across data-parallel ranks.
AdamWAdam optimiser with weight decay. Requires FP32 master weights + momentum + variance per parameter = 12 bytes/param in VRAM.
QLoRAQuantised Low-Rank Adaptation. 4-bit NF4-quantised frozen base + BF16 LoRA adapters. Enables fine-tuning of large models on consumer hardware.
LoRALow-Rank Adaptation. Adds trainable rank-decomposition matrices (A×B) to frozen layers. Merges into base model post-training.
BOMBill of Materials. Non-GPU costs of a server: chassis, motherboard, CPU, RAM, NVMe, networking, power supplies. Typically $9K–$18K+ per 8-GPU server.
PUEPower Usage Effectiveness. Data centre overhead ratio: PUE 1.25 means 25% of total power goes to cooling/UPS/lighting rather than IT load.
RAGRetrieval-Augmented Generation. LLM queries a vector database to retrieve relevant context chunks before generating a response.
MCPModel Context Protocol. Standard for exposing tools (APIs, databases, file systems) to LLMs via a structured protocol. Enables agentic tool use.
TTFTTime to First Token. Latency from request submission to the first output token being generated. Drives perceived responsiveness.
NVLinkNVIDIA's high-bandwidth GPU-to-GPU interconnect within a server. Required for Tensor Parallelism. H100/H200: 900 GB/s bidirectional.
InfiniBand NDR400 Gb/s network interconnect standard. Required for cross-node Pipeline Parallelism and Data Parallel gradient all-reduce at scale.
TaaSToken-as-a-Service. Compute monetisation model where customers buy prepaid token packs and draw down per inference call. Cost basis: GPU compute cost ÷ monthly token volume.
